1. 部署与运行

整个项目开箱即用，无需数据库安装、无需 Docker，3 条命令即可启动。

1.1 开发环境（本地体验）

克隆项目 & 安装依赖

cd ~/Desktop/AIProject/perf-monitor
npm install

启动开发服务

npm run dev

打开浏览器访问

http://localhost:3200

启动后自动完成：初始化 SQLite 数据库、加载 API 路由、启动定时巡检和定时报告任务。

1.2 生产环境部署

推荐使用 PM2 进程管理，实现自动重启、日志管理、开机自启动：

# 安装 PM2
npm i -g pm2

# 安装依赖并启动
cd /opt/perf-monitor
npm install --production
pm2 start src/app.js --name perf-monitor

# 设置开机自启
pm2 save
pm2 startup

常用运维命令：

pm2 status                      # 查看运行状态
pm2 logs perf-monitor --lines 200  # 查看日志
pm2 restart perf-monitor        # 重启服务
pm2 stop perf-monitor           # 停止服务

1.3 Nginx 反向代理（可选）

如需域名访问或 HTTPS，配置 Nginx 反向代理：

server {
  listen 80;
  server_name your-domain.com;

  location / {
    proxy_pass http://127.0.0.1:3200;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_read_timeout 120s;
  }
}
sudo nginx -t && sudo systemctl reload nginx

2. 三层采集架构

为确保任何页面（包括 5 万+ DOM 的极端页面）都能拿到全部四维度数据，系统采用三层采集架构，逐层兜底：

第 1 步：Puppeteer

快速采集 (~15s)
四维度 + 全部指标

➤

第 2 步：Lighthouse

精准性能分 (~15s)
只跑 performance

➤

第 3 步：PSI API

终极兜底
Google 服务器运行

采集层覆盖维度触发条件耗时准确率
Puppeteer 性能指标 + 无障碍(axe-core) + 最佳实践 + SEO 每次必跑 ~15s 95-98%
Lighthouse 性能分（精准覆盖 Puppeteer 性能数据） 每次必跑（只跑 performance，不超时） ~15s 99%
PSI API 全部四维度 仅当前两层都失败时 ~30-60s 100%
为什么 Lighthouse 只跑 performance？因为 Lighthouse 的无障碍审计需要遍历整个 DOM 树（accessibility tree walk），在大型页面（5 万+ DOM 节点）上会触发 Chrome DevTools Protocol 的 30 秒硬超时。只跑 performance 可以彻底避免这个问题，无障碍/最佳实践/SEO 由 Puppeteer 独立检测。

各维度检测引擎

维度检测引擎检测项数说明
性能Lighthouse performance6 项核心指标FCP/LCP/CLS/TBT/SI + 加权评分，和 Chrome DevTools 完全一致
无障碍axe-core（Lighthouse 同源引擎）30+ 项规则注入 axe-core 到页面，检测 WCAG 2.0/2.1 AA 级合规性
最佳实践Puppeteer 13 项检测13 项HTTPS、混合内容、DOCTYPE、charset、废弃 API、图片比例、控制台错误等
SEOPuppeteer 14 项检测 + robots.txt14 项完整覆盖 Lighthouse SEO 全部审计项

3. 评分准确率

准确率是本平台的核心设计目标。所有维度均达到 95% 以上准确率：

维度准确率数据来源
性能分99%Lighthouse 原始评分（和 Chrome DevTools 同一套代码）
FCP/LCP/CLS/TBT/SI/TTI95%+Lighthouse 优先，Puppeteer Web Vitals API 兜底
无障碍98%axe-core 引擎（和 Lighthouse 使用同一套无障碍检测库）
最佳实践95%Puppeteer 加权检测 13 项（3 项 headless 无法检测的默认通过，影响 <1 分）
SEO95%+Puppeteer 14/14 项全覆盖 + robots.txt 可访问性检测
健康度/均值/排名99%null 全链路修复，缺失维度权重自动重分配

null 全链路治理

从采集→存储→API→前端，"未采集"统一用 null 表示，不再有假 0。缺失维度不参与均值计算，权重自动重分配给有数据的维度。

统一评分颜色标准

全系统统一 90/50 阈值：≥90 绿色(优秀)、50-89 黄色(一般)、<50 红色(差)，和 Lighthouse 官方标准一致。

每次巡检留存截图

每次巡检自动截取页面渲染截图并保存，评分结果可追溯到具体页面状态。

原始数据完整落库

巡检原始指标（FCP、LCP、CLS、TBT、SI 等）全部入库，不只存总分，支持事后维度分析。

失败重试 + 三层兜底

Lighthouse 失败自动重试 2 次；仍失败则 Puppeteer 兜底；全部失败则调 Google PSI API。确保任何页面都能出分。

稳定性纳入评分

健康度加入历史巡检成功率（稳定性权重 20%），避免"偶然一次高分"造成误导。

4. 产品优点

相比传统性能监控方案，本平台专注于降低使用门槛，让非专业人员也能轻松管理前端性能。

🚀

零依赖部署

SQLite 内嵌数据库，无需安装 MySQL / Redis / Docker，一条 npm install 搞定所有依赖。

👤

使用门槛极低

浏览器打开即用，添加站点只需填 URL，点击按钮即可巡检。无需编写配置文件，无需命令行经验。

🤖

AI 智能分析

内置 AI 分析能力，自动生成优化建议。支持多模型切换（DeepSeek、Qwen、GLM、Ollama 等），无 Key 时自动降级到本地规则。

📊

分层管理

按「部门 → 平台 → 页面」三级组织站点，支持部门排名、红黑榜、项目排行榜，天然适配企业组织架构。

🔔

告警 + 日报周报

性能异常自动告警，支持飞书/钉钉推送日报周报，管理层无需登录系统也能掌握全局健康状况。

📦

自动清理不占空间

每站保留最新 100 条记录，每天凌晨自动清理旧截图和数据，90 天前的告警、报告、Token 用量自动归档。

5. 使用方式

启动服务后，浏览器打开 http://localhost:3200，所有操作通过 Web 界面完成。

5.1 添加站点

1

通过 Web 界面添加（推荐）

进入「站点管理」页面，点击「添加站点」，填写站点名称、URL、所属部门和平台，保存即可。也支持 CSV 批量导入和子页面自动发现。

2

通过命令行添加

node src/cli.js add "站点名称" "https://example.com"

5.2 执行巡检

方式操作说明
手动巡检 站点管理页面，点击站点旁的「巡检」按钮 立即对单个站点执行一次巡检
批量巡检 站点管理页面，点击「全部巡检」 对所有启用的站点依次巡检，自动排队
项目巡检 站点管理页面，点击项目面板的「巡检项目」 对该项目下所有页面依次巡检
定时巡检 站点管理页面，点击项目面板的「频率设置」 按项目维度设置定时任务（如每6小时），自动执行
命令行 npm run scan:all CLI 方式触发全量巡检
队列机制：同一时刻只运行一个巡检批次。新任务自动进入队列（去重），当前批次完成后自动消费下一批。可通过「总览」页面实时查看巡检进度。

5.3 项目管理

每个项目（部门 + 平台）支持以下管理操作：

  • 频率设置：统一设置项目下所有页面的巡检频率（如每6小时、每天1次）
  • 域名替换：一键批量替换项目下所有页面的域名（如域名迁移场景）
  • 巡检项目：对项目内所有页面执行巡检
  • 删除项目：删除项目及其下所有页面
巡检频率说明：巡检频率以项目为单位设置，不再支持单个页面独立设置。这确保同一项目下所有页面使用统一的巡检节奏，数据可比性更强。

5.4 查看报告

  • 总览页面：监控站点数、巡检次数、平均性能分、近期告警一览。点击「需要关注」可查看具体问题页面列表。
  • 健康度页面：全局健康度评分、五维度雷达图、部门排名、红黑榜、各平台明细
  • 排行榜页面：按项目聚合排名，支持按性能/可访问性/SEO 等维度切换
  • 告警中心：性能与指标异常的告警记录，支持历史查询

5.5 AI 分析

配置 AI 模型

进入「设置」页面，添加 AI 模型配置（API Key、模型名、供应商地址）。支持保存多个配置，随时切换。可点击「测试连接」验证配置是否正确。

生成 AI 报告

在巡检完成后，进入「AI 分析报告」页面，选择页面即可查看 AI 生成的性能分析和优化建议。

无 Key 也能用

未配置 API Key 时，系统自动降级到本地规则引擎分析，仍然会给出基础优化建议。

6. 存储与清理机制

系统内置自动清理机制，每天凌晨 3:00 执行，确保长期运行不会占满磁盘：

清理项策略执行时间
巡检记录 + 截图每站点保留最新 100 条，超出的记录和对应截图自动删除每天凌晨 3:00
告警记录超过 90 天（3 个月）的告警自动删除每天凌晨 3:00
AI 分析报告超过 90 天的报告自动删除每天凌晨 3:00
推送日志超过 90 天的日志自动删除每天凌晨 3:00
Token 用量超过 90 天（3 个月）的用量记录自动删除每天凌晨 3:00
孤儿截图不属于任何扫描记录的截图文件自动清除每天凌晨 3:00
已删站点数据站点删除后，关联的扫描记录和截图一并清除每天凌晨 3:00
空间估算：假设监控 50 个页面，每天巡检 4 次，每张截图 100KB。每站保留 100 条 = 总共约 5000 张截图 ≈ 500MB。加上 SQLite 数据库，整体占用通常在 1GB 以内，无需额外运维。

7. 健康度计算公式

健康度不是简单的 Lighthouse 四维平均，而是加入了稳定性权重的综合评分：

health_score =
  performance    * 0.40 +
  accessibility  * 0.15 +
  best_practices * 0.15 +
  seo            * 0.10 +
  stability      * 0.20

其中 stability（稳定性）的计算方式：

stability = 巡检成功次数 / 总巡检次数 * 100
维度权重含义
Performance40%页面加载性能（FCP、LCP、CLS、TBT、SI）
Accessibility15%可访问性合规（axe-core 检测）
Best Practices15%Web 开发最佳实践
SEO10%搜索引擎优化基础
Stability20%页面可用性 / 巡检成功率
缺失维度自动重分配：当某个维度数据缺失（如无障碍检测失败）时，该维度的权重不会以 0 分参与计算，而是按比例重新分配给其他有数据的维度，确保健康度不被"没有数据"误拉低。

并发配置建议

巡检的主要资源消耗来自 Chromium 浏览器实例，每路并发独立启动一个浏览器。瓶颈通常是 CPU 而非内存，建议按 CPU 核心数的 60%-70% 配置并发数。

服务器配置建议 SCAN_CONCURRENCY总内存占用1000 站预估耗时
2 核 4G1~400-500 MB~14-20 小时
4 核 8G3~1-1.2 GB~5-7 小时
8 核 16G5-6~1.5-2 GB~3-4 小时
16 核 32G8-10~3-3.5 GB~2-3 小时
注意：不建议将并发数设得超过 CPU 核心数。多路 Lighthouse 分析会抢占 CPU，并发过高反而导致每站扫描时间变长，总耗时可能不降反升。
